同意統合

このガイドでは、Tealium iQタグ管理システムとサポートされているベンダー統合を構成し、ユーザーが同意を与えた場合にのみタグが発火するように構成を確認する方法を説明します。

このガイドでは、Tealium iQタグ管理システムとOneTrust同意統合を使用しますが、この構成はサポートされているベンダー統合のいずれにも適用できます。このガイドの終わりまでに、OneTrustまたは他の任意のサポートされているConsent Management Platform（CMP）を成功裏に統合し、Tealium iQでユーザーの構成を尊重するための関連する強制措置を構成します。

Tealium iQでOneTrust同意統合を構成するには、以下の手順を完了します：

  1. CMPスクリプトを読み込む
  2. 統合を構成する
  3. 強制ルールを構成する
  4. 公開場所を選択する
  5. 目的グループを作成する
  6. 目的グループを編集する
  7. Tealium iQを目的に割り当てる
  8. タグを目的にマッピングする
  9. 構成を公開する

さらに、以下の手順で構成を確認し、トラブルシューティングを行います：

  1. データレイヤーを確認する
  2. ブラウザコンソールから同意統合構成を確認する
  3. 追加情報を表示するためにデバッグモードを使用する

要件

同意統合を作成するには、以下が必要です：

  • Tealium iQタグ管理
  • Tealium同意統合

統合を構成するには、以下のベンダー固有の情報が必要です：

  • ベンダーID：同意統合がページ上のCMPを識別するために使用します。
  • 構成済みの同意カテゴリ：同意統合はこれらを目的と呼び、1つのCMPからの目的の集合を目的グループと呼びます。

この情報は、すべてのトラッキングを受け入れた後のウェブページから直接取得できます。取得方法の詳細については、サポートされているベンダー統合を参照してください。

CMPスクリプトを読み込む

まず、Pre LoaderまたはDOM Ready拡張機能を使用して、OneTrust CMPスクリプトをウェブサイトに注入します。この構成には、CMPプロバイダから取得できる構成済みのCMPスクリプトが必要です。この例では、OneTrustコンソールから取得したOneTrust CMPスクリプトを使用します。

  1. iQタグ管理 > 拡張機能に移動します。
  2. + 拡張機能を追加 > 高度な構成をクリックします。
  3. Javascriptコード拡張機能を追加します。
  4. 拡張機能を識別するためのタイトルを入力します。
  5. スコープの下で、Pre LoaderまたはDOM Ready Extensionsを選択します。
  6. 構成の下に、OneTrust CMPスクリプトを貼り付けます： 
    window.addCmp = function () {
(function() {
    var domainId = '018dd272-a3f7-79cb-bf7a-b78b47dacb46-test';

    var head = document.getElementsByTagName('head')[0];
    var script = document.createElement('script');
    script.type = 'text/javascript';
    script.src = 'https://cdn.cookielaw.org/scripttemplates/otSDKStub.js';
    script.setAttribute("data-domain-script", domainId);
    //script.setAttribute("data-domain-script", b['OTDomainGUID']);
    script.async = true;
    head.appendChild(script);    
})();

次に、ユーザーが提供する同意に基づいてのみタグが発火するように統合を構成します。

Tealium iQでOneTrust同意統合を構成する

ステップ1：統合を構成する

  1. iQタグ管理 > 同意統合セクションに移動します。
  2. 統合を追加をクリックします。
  3. 統合の目的を明確に識別する名前を入力します。
  4. ベンダードロップダウンリストからOneTrustを選択します。
  5. OneTrustコンソールから取得したベンダーIDを入力します。
  6. OneTrustのデフォルトカテゴリーから標準的な目的を自動的に作成するには、OneTrustデフォルトカテゴリーで新しい目的グループを作成を選択します。
  7. 次へをクリックします。

ステップ2：強制ルールを構成する

強制ルールは、CMPによってキャプチャされた同意信号がいつ強制されるべきかを決定します。同意が常に正しく強制されるようにするために、ウェブサイト上の各ページビューやイベントは、重複がないように1つの同意統合または免除をトリガーする必要があります。

  1. 単一の統合のみを構成しているため、ドロップダウンリストからすべてのページとイベントを選択します。
  2. 次へをクリックします。

ステップ3：公開場所を選択する

公開場所画面では、統合が適用される環境を選択できます。まずテスト環境に公開することをお勧めします。この例では、すべての環境に公開します。

  1. Dev、QA、およびProd環境を選択します。
  2. 次へをクリックします。

ステップ4：目的グループを選択する

  1. 目的グループ画面で、ドロップダウンリストからOneTrustデフォルトを選択します。
  2. 同意統合を保存するために保存をクリックします。

ステップ5：目的グループを編集する

同意統合を保存した後、目的グループを編集してTealium iQを目的に割り当て、タグを目的にマッピングする必要があります。

目的グループを編集するには：

  1. 同意統合画面で、目的グループタブをクリックします。
  2. ステップ1で作成したOneTrustデフォルト目的グループを選択します。

ステップ6：Tealium iQを目的に割り当てる

Tealium iQ目的画面では、Tealium iQを特定の目的にマッピングできます。これは、すべてのタグ操作を管理するために重要です。同意が必要で、ユーザーがその目的に同意しない場合、Tealium iQはロードされず、結果としてタグは機能しません。

  1. Tealium iQ目的タブをクリックします。
  2. Tealium iQが常にロードされるようにするため、Tealium iQを厳密に必要な目的に割り当てます。

ステップ7：タグを目的にマッピングする

タグのマッピング画面では、タグを特定の目的に割り当てることができ、ユーザーの同意選択に基づいて正しいタグが発火することを保証します。すべてのタグに目的を割り当てることが重要で、目的がないタグは発火しません。この画面では、同意が同じイベント内で更新された場合にタグを再発火させる構成もできます。

  1. タグのマッピングタブをクリックします。
  2. 各タグを適切な目的に割り当てます。
  3. 同意が同じイベント内で更新された場合に、Tealium Collectタグなどのタグを再発火させたい場合は、タグの再発火を有効にします。
  4. 目的グループを保存するために保存をクリックします。

ステップ8：構成を公開する

Tealium iQプロファイルを保存し公開して、CMPとの同意統合を開始します。

これで、Tealium iQ内のOneTrust統合の構成が完了しました。この構成により、すべてのページとイベントで、CMPで構成した構成に従って、ユーザーが同意を与えた場合にのみタグが発火します。選択した環境でテストを行い、統合が正しく動作していることを確認します。テスト方法についての詳細は、以下のテストとトラブルシューティングを参照してください。

テストとトラブルシューティング

同意統合を構成したので、動作を確認しましょう。このセクションでは、Tealium iQのOneTrust同意統合が正しく機能していることを確認します。データレイヤーで同意状態を確認し、ブラウザコンソールを使用して構成を検証します。

開始する前に、以下を確認してください：

  • Tealium iQとOneTrust同意統合がウェブサイトで稼働しています。
  • ブラウザの開発者コンソールにアクセスできます。

データレイヤーを確認する

同意状態は、タグが発火する直前にデータレイヤーに追加されます。同意状態が正しくロードされていることを確認するために、tciプレフィックスのTealium同意変数を確認します：

  • tci.consent_type：同意の種類（暗黙的または明示的）を識別します。
  • tci.purposes_with_consent_all：すべての同意された目的を含む配列を含みます。
  • tci.purposes_with_consent_unprocessed：すべての未処理の同意された目的を含みます。
  • tci.purposes_with_consent_processed：すべての処理済みの同意された目的を含みます。

未処理と処理済みの変数は、EventStreamとAudienceStreamコネクタが同じイベントに対して複数回発火しないようにするロジックを構築するために使用できます。

同意統合構成の確認

同意統合がアクティブになった後、ブラウザのコンソールから直接現在の同意統合の詳細にアクセスできます。tealiumCmpIntegrationウィンドウスコープオブジェクト内には、統合の管理と対話のための関数が用意されています。

Webページから関連情報を取得するには、次の手順を実行します：

  1. CMPが実装されているウェブサイトを訪れます。

  2. すべてのトラッキングを承認します。

  3. 開発者ツールのJavaScriptコンソールを開きます。

  4. 次のスニペットをブラウザのコンソールに貼り付けます。

    var outputString = `${tealiumCmpIntegration.cmpName} - ${tealiumCmpIntegration.cmpCheckIfOptInModel() ? 'Opt-in' : 'Opt-out'} Model

Checks:
- vendor id:            ${tealiumCmpIntegration.cmpFetchCurrentLookupKey()}
- well-formed decision: ${tealiumCmpIntegration.cmpCheckForWellFormedDecision(tealiumCmpIntegration.cmpFetchCurrentConsentDecision())}
- explicit decision:    ${tealiumCmpIntegration.cmpCheckForExplicitConsentDecision(tealiumCmpIntegration.cmpFetchCurrentConsentDecision())}
- consented purposes:   ${JSON.stringify(tealiumCmpIntegration.cmpConvertResponseToGroupList(tealiumCmpIntegration.cmpFetchCurrentConsentDecision()).sort(),null, 8)}
`
console.log(outputString)

デバッグモードを使用して追加情報を表示

同意統合は、Tealiumのデバッグモードにも追加情報を追加します。

  1. デバッグモードを有効にして、追加の同意統合情報を表示します。
  2. デバッグモード出力で同意統合情報を簡単に見つけるために、正規表現/SENDING|****/を使用して出力をフィルタリングし、タグ送信と同意統合通知のみを表示します。

デバッグモードフィルタを使用すると、重要な同意関連メッセージを隔離し、次のことが可能になります：

  • 同意選択に基づくタグの動作をすばやく確認：フィルタは、ユーザーの同意に従って送信またはブロックされるタグを強調表示し、構成がユーザーの構成を尊重していることを確認します。
  • 潜在的な構成問題を簡単に特定：関連するログエントリに焦点を当てることで、フィルタは目的のマッピングの欠落や意図しないタグのブロックなどの問題を見つけ出し、それらをより効率的に解決できるようにします。

以下は、デバッグモードフィルタで遭遇する可能性があるいくつかのシナリオです。

シナリオ1：CMPスクリプトのブロック

このシナリオでは、CMPスクリプトがブロックされているか利用できません。デバッグ出力に示されているように、同意統合はTealium iQのロードを防ぎ、アクティブな同意統合が存在するため、ページビューイベントがキューに入れられますが、CMPスクリプトがロードされるまで他のアクションは発生しません。

シナリオ2：クッキーが承認される前のopt-inモードのCMP

ここでは、CMPはopt-inモードで動作しており、ユーザーはまだ明示的な同意を提供していません。ユーザーがクッキーを承認する前に、同意統合はNecessaryにマッピングされた目的のみに対する暗黙的な同意を検出し、他の目的のロードを防ぎます。opt-inモードにあるため、システムは明示的な同意決定を求めて継続的にポーリングします。

シナリオ3：クッキーが承認された後のopt-inモードのCMP

このシナリオでは、ユーザーがクッキーに同意した後、同意統合は明示的な同意を検出し、新たに同意された目的をレビューし、暗黙的な同意の下で既に処理されたものを除外します。次に、新たに同意された目的にマッピングされたタグを検索し、それらを発火のためにキューに入れます。目的にマッピングされていないタグは、ロードがブロックされたままです。

詳細については、同意統合の検証とデバッグを参照してください。
