同意統合

このガイドでは、Tealium iQタグ管理システムでサポートされているベンダー統合を構成し、ユーザーが同意を与えた場合にのみタグが発火するように構成を確認する方法について説明します。

このガイドでは、Tealium iQタグ管理システムとOneTrust同意統合を使用しますが、この構成はサポートされているベンダー統合のいずれにも適用できます。このガイドの終わりまでに、OneTrustまたは他のサポートされている同意管理プラットフォーム（CMP）を選択し、Tealium iQで関連する施行を統合して、タグがユーザーの構成を尊重するように構成します。

Tealium iQでOneTrust同意統合を構成するために、以下のステップを完了してください：

  1. CMPスクリプトをロードする
  2. 統合を構成する
  3. 施行ルールを構成する
  4. 公開場所を選択する
  5. 目的グループを作成する
  6. 目的グループを編集する
  7. Tealium iQを目的に割り当てる
  8. タグを目的にマッピングする
  9. 構成を公開する

さらに、以下のステップで構成を検証およびトラブルシューティングします：

  1. データレイヤーをチェックする
  2. ブラウザコンソールから同意統合構成を検証する
  3. 追加情報を表示するためにデバッグモードを使用する

必要条件

同意統合を作成するには、以下が必要です：

  • Tealium iQタグ管理
  • Tealium同意統合

統合を構成するためには、以下のベンダー固有の情報が必要です：

  • ベンダーID：同意統合がページ上のCMPを識別するために使用されます。
  • 構成された同意カテゴリ：同意統合はこれらを目的として参照し、1つのCMPからの目的の集合を目的グループと呼びます。

この情報は、すべてのトラッキングを受け入れた後に直接ウェブページから取得できます。それを取得する方法の詳細については、サポートされているベンダー統合を参照してください。

CMPスクリプトをロードする

始めに、Pre LoaderまたはDOM Ready拡張機能を使用して、ウェブサイトにOneTrust CMPスクリプトを注入します。この構成には、CMPプロバイダーから取得した構成済みのCMPスクリプトが必要です。この例では、OneTrustコンソールから取得したOneTrust CMPスクリプトを使用します。

  1. iQタグ管理 > 拡張機能に移動します。
  2. + 拡張機能を追加 > 上級をクリックします。
  3. JavaScriptコード拡張機能を追加します。
  4. 拡張機能を識別するためのタイトルを入力します。
  5. スコープの下で、Pre LoaderまたはDOM Ready拡張機能を選択します。
  6. 構成の下で、OneTrust CMPスクリプトを貼り付けます： 
    // ベンダーCMPスクリプトをロードする関数を定義する
window.addCmp = function () {
  var domainId = '12345678-1234-1234-1234-1234567890ab-test';

  var head = document.getElementsByTagName('head')[0];
  var script = document.createElement('script');
  script.type = 'text/javascript';
  script.src = 'https://cdn.cookielaw.org/scripttemplates/otSDKStub.js';
  script.setAttribute("data-domain-script", domainId);
  // script.setAttribute("data-domain-script", b['OTDomainGUID']);
  script.async = true;
  head.appendChild(script);    
};

// スクリプトをロードするためにすぐに'addCmp'関数を呼び出す
window.addCmp();

次に、ユーザーが提供する同意に基づいてのみタグが発火するように統合を構成します。

Tealium iQでOneTrust同意統合を構成する

ステップ1: 統合を構成する

  1. iQタグ管理 > 同意統合セクションに移動します。
  2. 統合を追加をクリックします。
  3. 統合の目的を明確に識別する名前を入力します。
  4. ベンダーのドロップダウンリストからOneTrustを選択します。
  5. OneTrustコンソールから取得したベンダーIDを入力します。
  6. OneTrustの標準目的から自動的に標準目的を作成する場合は、OneTrust Default Categoriesで新しいPurpose Groupを作成するを選択します。
  7. 次へをクリックします。

ステップ2: 施行ルールを構成する

施行ルールは、CMPによってキャプチャされた同意信号が施行されるタイミングを決定します。同意が常に正しく施行されるようにするために、ウェブサイトの各ページビューまたはイベントは、重複なしに1つの同意統合または例外をトリガーする必要があります。

  1. ドロップダウンリストからすべてのページとイベントを選択します。私たちは単一の統合のみを構成しています。
  2. 次へをクリックします。

ステップ3: 公開場所を選択する

公開場所画面では、統合が適用される環境を選択できます。最初にテスト環境に公開してからライブにすることをお勧めします。この例では、すべての環境に公開します。

  1. Dev, QA、およびProd環境を選択します。
  2. 次へをクリックします。

ステップ4: 目的グループを選択する

  1. Purpose Group画面で、ドロップダウンリストからOneTrust Defaultを選択します。
  2. 同意統合を保存するために保存をクリックします。

ステップ5: 目的グループを編集する

同意統合を保存した後、Tealium iQを目的に割り当ててタグを目的にマッピングするために、目的グループを編集する必要があります。

目的グループを編集するには：

  1. 同意統合画面で、Purpose Groupsタブをクリックします。
  2. ステップ1で作成したOneTrust Default目的グループを選択します。

ステップ6: Tealium iQを目的に割り当てる

Tealium iQ Purpose画面では、Tealium iQを特定の目的にマッピングできます。これは、すべてのタグ操作を管理するために重要です。同意が必要であり、ユーザーがその目的に同意しない場合、Tealium iQはロードされず、結果としてタグは機能しません。

  1. Tealium iQ Purposeタブをクリックします。
  2. Tealium iQが常にロードされるように、Strictly Necessary目的にTealium iQを割り当てます。これにより、ユーザーはオプトアウトできません。

ステップ7: タグを目的にマッピングする

Map Tags画面では、特定の目的にタグを割り当てることができ、ユーザーの同意選択に基づいて正しいタグが発火するようにします。すべてのタグに目的を割り当てることが重要です。目的が割り当てられていないタグは発火しません。この画面では、同じイベント内でユーザーの同意が更新された場合にタグを再発火させることも構成できます。

  1. Map Tagsタブをクリックします。
  2. 各タグを適切な目的に割り当てます。
  3. Tealium Collectタグなどのタグが同じイベント内で同意が更新された場合に再発火するように、タグの再発火を有効にします。
  4. 目的グループを保存するために保存をクリックします。

ステップ8: 構成を公開する

Tealium iQプロファイルを保存して公開し、CMPとの同意統合を使用を開始します。

これで、Tealium iQ内でOneTrust統合が正常に構成されました。この構成により、CMPで構成した構成に従って、すべてのページとイベントでユーザーの同意が与えられた場合にのみタグが発火します。選択した環境でテストして、統合が正しく機能していることを確認してください。テスト方法についての詳細は、以下のテストとトラブルシューティングを参照してください。

テストとトラブルシューティング

同意統合を構成したので、それが機能していることを確認しましょう。このセクションでは、Tealium iQのOneTrust同意統合が正しく機能していることを検証します。データレイヤーで同意状態をチェックし、ブラウザコンソールを使用して構成を検証します。

始める前に、以下を確認してください：

  • Tealium iQとOneTrust同意統合がウェブサイト上で稼働しています。
  • ブラウザの開発者コンソールにアクセスできます。

データレイヤーを確認する

同意状態は、タグが発火する直前にデータレイヤーに追加されます。同意状態が適切にロードされているかを確認するために、tciプレフィックスを持つTealiumの同意変数を確認してください：

  • tci.consent_type: 同意のタイプ（暗黙または明示）を識別します。
  • tci.purposes_with_consent_all: 同意された目的の配列を含みます。
  • tci.purposes_with_consent_unprocessed: 未処理の同意された目的を含みます。
  • tci.purposes_with_consent_processed: 処理済みの同意された目的を含みます。

未処理および処理済みの変数は、EventStreamおよびAudienceStreamコネクタが同じイベントに対して複数回発火しないようにするロジックを構築するために使用できます。

同意統合セットアップを確認する

同意統合がアクティブになった後、ブラウザコンソールから直接現在の同意統合の詳細にアクセスできます。tealiumCmpIntegrationウィンドウスコープオブジェクト内には、統合を管理および操作するための関数が用意されています。

ウェブページから関連情報を取得するには、以下の手順に従ってください：

  1. CMPが実装されているウェブサイトを訪問します。

  2. すべてのトラッキングを受け入れます。

  3. 開発者ツールのJavaScriptコンソールを開きます。

  4. 次のスニペットをブラウザコンソールに貼り付けます。

    var outputString = `${tealiumCmpIntegration.cmpName} - ${tealiumCmpIntegration.cmpCheckIfOptInModel() ? 'Opt-in' : 'Opt-out'} Model

Checks:
- vendor id:            ${tealiumCmpIntegration.cmpFetchCurrentLookupKey()}
- well-formed decision: ${tealiumCmpIntegration.cmpCheckForWellFormedDecision(tealiumCmpIntegration.cmpFetchCurrentConsentDecision())}
- explicit decision:    ${tealiumCmpIntegration.cmpCheckForExplicitConsentDecision(tealiumCmpIntegration.cmpFetchCurrentConsentDecision())}
- consented purposes:   ${JSON.stringify(tealiumCmpIntegration.cmpConvertResponseToGroupList(tealiumCmpIntegration.cmpFetchCurrentConsentDecision()).sort(),null, 8)}
`
console.log(outputString)

    いくつかのカテゴリに同意しましょう。ページをリロードした後、再びCMPとその運用モードを確認できます。今回は明示的な同意が与えられていますが、C0001（必要）およびC0002（パフォーマンス）目的のみが許可されています。

デバッグモードを使用して追加情報を表示する

同意統合はTealiumのデバッグモードにも追加情報を提供します。

  1. デバッグモードを有効にして、同意統合情報を表示します。
  2. デバッグモード出力で同意統合情報を簡単に見つけるために、正規表現 /SENDING|****/ を使用して出力をフィルタリングし、タグ送信と同意統合通知のみを表示します。

デバッグモードフィルタを使用することで、次のことが可能になります：

  • 同意選択に基づいたタグの動作を迅速に確認：フィルタは、ユーザーの同意に応じて送信またはブロックされるタグを強調表示し、セットアップがユーザーの構成を尊重していることを保証します。
  • 潜在的な構成問題を簡単に特定：関連するログエントリに焦点を当てることで、目的マッピングの欠落や意図しないタグブロッキングなどの問題を発見し、効率的に解決できます。

デバッグモードフィルタを使用したいくつかのシナリオを以下に示します。

シナリオ1: CMPスクリプトのブロック

このシナリオでは、CMPスクリプトがブロックされているか利用できません。デバッグ出力に示されているように、同意統合が有効であるため、Tealium iQのロードが防止されています。ページビューイベントはキューに入れられますが、CMPスクリプトがロードされるまで他のアクションは発生しません。

シナリオ2: クッキーが受け入れられる前のオプトインモードのCMP

ここでは、CMPがオプトインモードで運用されており、ユーザーはまだ明示的な同意を提供していません。ユーザーがクッキーを受け入れる前に、同意統合はNecessaryにマップされた目的のみの暗黙の同意を検出し、他の目的のロードを防ぎます。オプトインモードでは、システムは明示的な同意決定を継続的にポーリングします。

シナリオ3: クッキーが受け入れられた後のオプトインモードのCMP

このシナリオでは、ユーザーがクッキーに同意した後、同意統合は明示的な同意を検出し、新たに同意された目的を確認しますが、暗黙の同意の下で既に処理された目的は除外します。その後、新たに同意された目的にマップされたタグを発火するためにキューに入れます。目的にマップされていないタグは引き続きロードがブロックされます。

詳細については、同意統合の検証とデバッグを参照してください。
