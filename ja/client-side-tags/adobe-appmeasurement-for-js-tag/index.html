Adobe Analytics AppMeasurement for JS タグの構成ガイド

この記事では、iQ Tag ManagementアカウントでAdobe Analytics AppMeasurement for JSタグを構成する方法について説明します。

タグのヒント

  • Adobe Experience Cloud IDサービスを使用するには、このタグを追加する前にAdobe Experience Cloud IDサービスタグを追加してください。
  • AppMeasurementタグでは、すべてのレガシーH26の機能がサポートされているわけではありません。
  • サーバーの値を正しく入力してください。たとえば、サードパーティのサーバーでは 112.2o7.net または 122.2o7.net などがあります。
  • モバイルでAppMeasurementを使用するには、アプリケーションにTealium Mobile SDKを実装する必要があります。
  • サポートされているプラグインのリストについては、AppMeasurementプラグインサポートを参照してください。
  • 追加の情報については、JavaScript用AppMeasurementについてを参照してください。

タグの構成

タグマーケットプレイスに移動し、Adobe Analytics AppMeasurement for JSタグを追加します（タグの追加方法について詳しくはこちら）。

タグを追加した後、次の構成を行います。

  • コードバージョン
    • AppMeasurement.js ライブラリのバージョンです。
  • デバイスタイプ
    • 実装プラットフォームに基づいて次のいずれかを選択します：
      • デスクトップウェブサイトのインストールには Standard を選択します。
      • Tealium for iOS または Tealium for Android を使用したネイティブモバイルのインストールには Mobile App を選択します。
  • レポートスイート
    • 必須です。
    • 動的にレポートスイートを構成するためのマッピングを使用している場合でも、デフォルト値を構成してください。
    • 使用するデフォルトのレポートスイート。Dynamic Acct Listで構成された値に基づいて変更される場合があります。
    • AppMeasurement.jsを参照： s.account
  • サーバー
    • 必須です。
    • データ収集サーバーです。
    • AppMeasurement.jsを参照： s.trackingServer
    • たとえば、metrics.tealium.com または tealiumclient.122.2o7.net などです。
  • セキュアサーバー
    • 必須です。
    • https データ収集サーバーです。
    • AppMeasurement.jsを参照： s.trackingServerSecure
    • たとえば、smetrics.tealium.com または tealiumclient.122.2o7.net などです。
  • 自動イベントシリアル化の検出
    • トリガー文字列内でイベントのシリアル化を自動的に検出します。
    • イベントトリガー値にコロン（:）が含まれる場合は、No に構成する必要があります。
  • 自動リンクトラッキング
    • デフォルト値は Yes で、推奨される選択肢です。
    • No に構成する場合、Link Tracking Extension などの他の方法を使用して自動リンクトラッキングを再現する必要があります。
  • 内部リンクフィルター
    • カンマ区切りのドメインのリストで、内部として分類されるリンククリックがAdobeレポートに出口クリックとして報告されないようにします。
    • AppMeasurement.jsを参照： s.linkInternalFilters
    • たとえば、tealium.com,tealiumiq.com などです。
  • clearVarsを実行
    • デフォルト値は No です。
    • 各トラッキングリクエスト後にグローバルSオブジェクトで構成されたprops、eVars、およびイベントをクリアします。
    • Data Mapping スクリーンでは、clearVars_in_RPTCallback パラメータを使用して registerPostTrack コールバック内で clearVars アクションを割り当てます。
  • S-Object名
    • 必須です。
    • デフォルト名は s です。
    • ページ上で複数のインスタンスのAdobe AnalyticsまたはAppMeasurementを実行している場合は、異なる名前を構成してください。
  • 名前空間
    • オプションです。
    • 訪問の名前空間です。
    • サーバーとセキュアサーバーが 2o7.net で終わる場合にのみ適用されます。
    • AppMeasurement.jsを参照： s.visitorNamespace
  • Experience Cloud ID
    • オプションです。
    • Adobe Experience Cloud IDです。
    • バージョン1.3.x以降でサポートされているVisitor APIを使用している場合は、Enterprise Cloud IDをここに入力してください。
    • Adobe Enterprise Cloud ID Serviceについて詳しくはこちらをご覧ください。
  • Advertising Cloud Viewthroughを実行
    • バージョン2.17以降でのみ利用可能です。
    • Advertising Cloud Viewthroughファイルをロードし、ビュースルーデータを送信します。
  • Viewthroughインターバル
    • ビュースループスクリプトをロードする間隔（ミリ秒単位）を構成します。
  • Viewthrough最大試行回数
    • ビュースループをロードする前に最大試行回数を構成します。
  • タイムスタンプの送信
    • 各トラッキングコールと共にタイムスタンプを送信することを許可します。モバイルデバイストラッキングのみに適用されます。

ロードルール

ロードルール は、サイト上のこのタグのインスタンスをいつ、どこにロードするかを決定します。

推奨されるロードルール：すべてのページ。

マッピングは、データレイヤー変数 のデータをベンダータグの対応する変数に送信するプロセスです。変数をタグの宛先にマッピングする方法についての手順は、データマッピングを参照してください。

利用可能なカテゴリは次のとおりです：

  • この宛先でマッピングを使用している場合、タグの構成で構成されたレポートスイートがオーバーライドされます。

  • 必要に応じて実装を標準化するための簡略化された方法です。たとえば、異なるサイトからデータを組み合わせる場合などに使用できます。

  • デフォルトは false です。

イベント

イベントのマッピングは、他の宛先のマッピングとは異なります。

イベントをマッピングするには、次の手順を使用します：

  1. データマッピング で、Variables ドロップダウンリストからイベント変数を選択します。
  2. Select Destination をクリックし、Event Triggers カテゴリに移動します。
  3. Value フィールドに、マッピングする変数の値を入力します。
    この変数がトリガー文字列になります。
  4. Trigger ドロップダウンリストから、トリガーするイベントを選択します。
    追加の値/トリガーの組み合わせを追加するには、プラスアイコン（+）をクリックします。
宛先名 説明
Value
  • 割り当てられたイベントをトリガーするマッピングされた変数の値。
Trigger
  • トリガーするイベント。
  • 次を選択します：
    • 製品ビューの場合は prodView。
    • 新しいショッピングカートの開始/初期化の場合は scOpen。
    • 商品をショッピングカートに追加する場合は scAdd。
    • ショッピングカートから商品を削除する場合は scRemove。
    • ショッピングカートを表示する場合は scView。
    • チェックアウトを開始する場合は scCheckout。
    • 注文を完了する場合は purchase。
    • カスタムイベントを構成する場合は event1 から event1000。
      注：イベント101〜1000はライブラリバージョン1.4以降でのみ使用できます。
      • SiteCatalystでは、通常値を含まない特定のサイトで発生するイベントを100個使用できます。これらのイベントは、ログインやフォームビューなどの特定のサイトで発生する特定の事柄を通常数えるために使用されます。

製品レベルのイベント

SiteCatalystでは、製品固有のアクションを送信するための100の宛先が用意されています。

宛先名 説明
PRODUCTS_event1 から PRODUCTS_event100 割り当てる製品イベント。

値イベント

SiteCatalystでは、値イベントを送信するための100の宛先が用意されています。

宛先名 説明
VALUE_event1 から VALUE_event100 割り当てる値イベント。

Props

SiteCatalystでは、カスタムプロップ（トラフィック変数）を使用できます。

宛先名 説明
PRODUCTS_event1 から PRODUCTS_event100
  • 割り当てる製品イベント。
  • 75以上のプロップはマッピングツールボックスに組み込まれていません。

SiteCatalystでは、カスタムプロップ（トラフィック変数）を使用できます。プロップには、ページビューを超えて持続するデータが含まれており、プロップと他の変数を同じページで構成する必要があります。数値を保持するためにイベントを使用することもできます。

カスタム宛先をマッピングする手順は次のとおりです。

  1. 変数を任意のプロップ宛先にマッピングします。
  2. 上部のテキストボックスに、選択した組み込み宛先の代わりにカスタム宛先を入力します。
  3. 他の変数をマッピングするか、Done をクリックして終了します。

  • SiteCatalystでは、250のeVarパラメータ（変換変数とも呼ばれる）を使用できます。
  • eVarには、単一のページビューを超えて持続するデータが含まれており、eVarから他の変数にデータを関連付けることができます。レポートの構成方法によっては、eVarから他の変数にデータを関連付けることができます。
eVarには、内部検索用語、A/Bテスト、キャンペーン/プロモーションのトラッキング、商品カテゴリ、ユーザータイプなどが一般的に含まれます。

コマース

AppMeasurementタグは、Eコマース対応であり、デフォルトのE-Commerce Extensionのマッピングを自動的に使用します。通常、このカテゴリで手動でマッピングする必要はありません。ただし、拡張マッピングまたは所望のEコマース変数が拡張で提供されていない場合には、手動でマッピングする必要があります。

Eコマース拡張によって、製品文字列などの情報が自動的に入力されます。SiteCatalystは、製品情報、収益、販売数量などの情報を製品文字列でキャプチャします。標準の製品文字列は次のようになります：

製品文字列のベストプラクティス：
  • セミコロン ; は、製品文字列のほとんどの項目の標準の区切り記号です。
  • 複数のマーチャンダイジングeVarは、パイプ（|）文字で区切ります。
  • 複数の製品インスタンスは、カンマ（,）で区切ります。

イベントシリアル化の検出

AppMeasurementタグは、データレイヤーのトリガー値内でのイベントシリアル化を自動的に検出します。タグが発火すると、タグはデータマッピングのEventsタブでマッピングされたトリガー文字列とデータレイヤーのトリガー値を照合し、どちらかのトリガーがコロンを含んでいるかどうかをチェックします。

  • イベントシリアル化の検出：オン
    イベントシリアル化の検出がオンになっている場合（デフォルト構成）、タグはコロンの隣にあるトリガー値を検出し、残りの文字列をマッピングされたトリガーと一致させようとします。完全一致がある場合、イベントはシリアル化されます。
  • イベントシリアル化の検出：オフ
    イベントシリアル化の検出がオフになっている場合、タグはコロンを検出しません。タグは、マッピングされたトリガーの両方を完全に一致させようとします。トリガー文字列にコロンが含まれる場合、イベントが正しくトリガーされる唯一の方法です。

イベントシリアル化の検出がオンまたはオフであっても、Event Serializationタブで直接マッピングされたシリアルは常に優先されます（シナリオ3の例を参照）。

次のシナリオは、コロンの有無に応じてトリガー値をどのように解釈するかを理解するのに役立ちます。

このセクション全体で、「マッピングされたトリガー」とは、タグのデータマッピング構成の値を指し、「データレイヤーのトリガー」とは、ウェブサイトのデータレイヤーで見つかる値を指します。

シナリオ1：データレイヤーのトリガーが fireEvt で、serial の値が空の場合

var utag_data = {
  trigger: "fireEvt",
  serial: ""
}
  • イベントシリアル化の検出：オン
    タグは fireEvt をマッピングされたトリガーと一致させ、ページ上で prodView を正常に発火しますが、シリアル化は行われません。
  • イベントシリアル化の検出：オフ
    上記と同じ結果です。

成功したネットワークコールは次のようになります：

s.events = "prodView";

シナリオ2：データレイヤーのトリガーが fireEvt:123 で、serial の値が空の場合

var utag_data = {
  trigger: "fireEvt:123",
  serial: ""
}
  • イベントシリアル化の検出：オン
    タグは 123 をシリアル化し、fireEvt をマッピングされたトリガーと一致させ、ページ上で prodView を正常に発火します。
    成功したネットワークコールは次のようになります：
s.events = "prodView:123";
  • イベントシリアル化の検出：オフ
    タグは fireEvt:123 をマッピングされたトリガーと一致させることができず、prodView はページ上で発火しません。

シナリオ3：データレイヤーのトリガーが fireEvt:123 で、serial の値が 456 の場合

var utag_data = {
  trigger: "fireEvt:123",
  serial: "456"
}
  • イベントシリアル化の検出：オン
    最初に、タグは 123 をシリアル化し、fireEvt をマッピングされたトリガーと一致させます。その後、prodView が正常に発火し、123 の代わりに 456 でシリアル化されます。
    これは、イベントシリアル化マッピングによって検出されたシリアル値（456）が、イベントシリアル化検出によって検出されたシリアル値（123）よりも優先されるためです。
    成功したネットワークコールは次のようになります：
s.events = "prodView:456";
  • イベントシリアル化の検出：オフ
    タグは fireEvt:123 をマッピングされたトリガーと一致させることができず、prodView はページ上で発火しません。

例：コロンを含むマッピングされたトリガー

この例は、シナリオ2と同じ経路をたどりますが、マッピングされたトリガーが fireEvt ではなく fireEvt:123 です。

var utag_data = {
  trigger: "fireEvt:123",
  serial: ""
}
  • イベントシリアル化の検出：オン
    タグは 123 をシリアル化しますが、マッピングされたトリガーの fireEvt と一致させることができません。そのため、prodView はページ上で発火しません。
  • イベントシリアル化の検出：オフ
    タグは fireEvt:123 をマッピングされたトリガーと一致させ、prodView が正常に発火します。このため、文字列にコロンが含まれる場合は、イベントシリアル化の検出をオフにすることをお勧めします。
